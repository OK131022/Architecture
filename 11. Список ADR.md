Список ADR (Architecture Decision Records) для спортивного приложения  
<br/>ADR (Architecture Decision Record) - это документ, описывающий важное архитектурное решение, принятое в проекте. Он включает в себя контекст проблемы, рассматриваемые варианты, принятое решение и обоснование выбора. Использование ADR помогает документировать архитектурные решения, облегчает понимание и поддержку системы, а также позволяет избежать повторного принятия решений по одним и тем же вопросам.  
<br/>Вот список ADR для спортивного приложения, охватывающий ключевые архитектурные аспекты:  
<br/>Общие решения:  

ADR-001. Выбор архитектурного стиля - Микросервисы  
Статус: Принято

Контекст: Требования к масштабируемости, гибкости, независимости развертывания и отказоустойчивости.

Решение: Использовать микросервисную архитектуру.

Обоснование: Микросервисы позволяют масштабировать каждый сервис независимо, использовать разные технологии для разных сервисов, обеспечивать отказоустойчивость и упрощать развертывание.

Последствия: Необходимость в Service Discovery, API Gateway, распределенных транзакциях и т.д.  
<br/>ADR-002: Database per Service  
Статус: Принято

Контекст: Необходимость независимой масштабируемости, гибкости в выборе БД и снижения связанности между сервисами.

Решение: Использовать принцип Database per Service.

Обоснование: DBS обеспечивает независимость, гибкость и отказоустойчивость.

Последствия: Сложность в реализации транзакций, охватывающих несколько сервисов.  
<br/>ADR-003: API Gateway (Внешний и Внутренний)  
Статус: Принято

Контекст: Необходимость централизованного управления API, аутентификации, авторизации, маршрутизации и rate limiting.  
Решение: Использовать два API Gateway: внешний (для клиентских приложений и внешних интеграций) и внутренний (для межсервисного взаимодействия).

Обоснование: Разделение API Gateway улучшает безопасность, масштабируемость и управляемость.

Последствия: Необходимость в настройке и управлении двумя API Gateway.  
<br/>ADR-004: Выбор протокола для межсервисного взаимодействия – gRPC

Статус: Принято

Контекст: Требования к высокой производительности, эффективности и надежности межсервисного взаимодействия.

Решение: Использовать gRPC для межсервисного взаимодействия.

Обоснование: gRPC использует Protocol Buffers для сериализации данных, что обеспечивает высокую производительность и эффективность. Он также поддерживает потоковую передачу данных и двунаправленную связь.

Последствия: Необходимость в изучении и использовании Protocol Buffers.  
<br/>ADR-005: Стратегия Мониторинга и Логирования

Статус: Принято

Контекст: Необходимость эффективного мониторинга, отладки и анализа проблем в распределенной системе.

Решение: Использовать ELK Stack (Elasticsearch, Logstash, Kibana) для централизованного логирования и Prometheus/Grafana для мониторинга метрик.  
Обоснование: ELK Stack - мощный инструмент для анализа логов, Prometheus/Grafana - для мониторинга производительности.  
Последствия: Необходимость настройки и управления инфраструктурой ELK Stack и Prometheus/Grafana.  
<br/>ADR-006. User_DB, Training_DB, Gamification_DB, Marketing_DB, Inventory_DB - Выбор базы данных

Статус: Принято

Контекст: Требования к надежному хранению данных о пользователях, аутентификации и авторизации, данных о тренировках, планах и активности пользователей, к хранению информации о товарах, запасах и использовании инвентаря, данных о баллах, значках и достижениях пользователей, о рекламных кампаниях, контенте и сегментации пользователей.

Решение: Использовать PostgreSQL.

Обоснование: PostgreSQL - реляционная база данных с открытым исходным кодом, надежная, масштабируемая и поддерживающая ACID транзакции.

Последствия: Необходимость в изучении и использовании SQL.  

ADR-007. Выбор Object Storage для хранения медиафайлов  
Статус: Принято

Контекст: Приложение должно хранить большое количество медиафайлов (фотографии, видео, аудио), загружаемых пользователями (аватары, фотографии тренировок, контент для социальных групп).  
Решение: Использовать Object Storage

Обоснование: Object Storage - объектное хранилище для хранения медиафайлов. Лучше всего подходит для хранения медиафайлов, учитывая требования к масштабируемости, надежности, экономичности и скорости доступа

Последствия: : Необходимость в изучении и использовании Object Storage  
<br/>ADR-008: Search_Service - Выбор технологии индексирования и поиска  
Статус: Принято

Контекст: Требования к быстрому и эффективному поиску по данным о пользователях, тренировках и инвентаре.

Решение: Использовать Elasticsearch.

Обоснование: Elasticsearch - мощная поисковая система, основанная на Lucene. Подходит для индексирования и поиска больших объемов данных.

Последствия: Необходимость в изучении и использовании Elasticsearch.  

ADR-009: Выбор Стратегии Контейнеризации и Оркестрации

Статус: Принято

Контекст: Необходимость в эффективном управлении контейнерами, масштабировании и развертывании сервисов.

Решение: Использовать Docker для контейнеризации и Kubernetes для оркестрации контейнеров.

Обоснование: Docker упрощает создание и развертывание контейнеров, а Kubernetes обеспечивает масштабирование микросервисной архитектурой.  
Последствия: Зависимость от выбранного облачного провайдера.  

ADR-010: Выбор облачного провайдера  
Статус: Предложено

Контекст: Необходимо выбрать облачного провайдера для размещения инфраструктуры приложения.  
Решение: Рассмотреть AWS, Azure и Google Cloud Platform (GCP) и выбрать наиболее подходящего провайдера на основе стоимости, функциональности и доступности.  
Обоснование: Все три провайдера предлагают широкий спектр услуг, необходимых для развертывания и управления микросервисной архитектурой.  
Последствия: Зависимость от выбранного облачного провайдера.
